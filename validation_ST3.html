Feedback page validation report

I initially tried to run the document, feedback.html, through the W3C Nu HTML Checker and got one error: a shared footer (the bottom of the document) contained an <h4> after an <h2>, which is not a logical hierarchy in order to be accessible. The footer was part of all four team pages, not just my page, and thus there was this same error on everyone's page. I did not wait on a group-wide template fix but changed the footer's <h4> tags to <h3> in my own file in order to get a clean pass by fixing the order of the headings. The page passed with no errors or warnings post this fix. I didn't have to leave any unanswered warnings, the validator gave me a clean bill of health after I corrected the heading problem.

Team page validation report

The W3C Nu HTML Checker found one error: A missing heading of <h2>; (should be used for a team member's name, for example "Yohan") comes after the page's <h1>, "Meet the Team" heading, and is not present in between. This occurred because there is only one level of heading below the main heading on this page; there is no "natural" level 2 heading to place without reworking the card structure. I chose not to force this warning to be an unnecessary <h2> since the names of team members are intended to be sub-items underneath a single page heading, not underneath an extra intermediate heading. The rest of the page passed without the appearance of any other errors, and I didn't believe that a heading was necessary, so I considered this one heading-level skip to be a good compromise.

Content page validation report

The content_ST3.html was checked for validation on the first pass and highlighted two problems. One was a misspelling in the CSS, with the correct spelling reading "balck" instead of "black". The other was the same incorrect heading level that was seen on the feedback page, an <h4> after an <h2> which meant there was no heading level between the two, this is not in line with the hierarchy required for accessibility. Temporarily changed the <h4> tags in the footer to <h3>, as I did on the feedback page, to restore the correct order of headings on that page, to correct the spelling mistake that was introduced; changed the same on my own copy, in my own file. Once both fixes were made the page had no remaining errors or warnings.

Reflection on Validation

Running my three pages through the validator taught me that the small errors are the ones that catch you off guard. The heading hierarchy warning on the shared footer was one of those - the page rendered perfectly fine in the browser but the semantic order was wrong because we all inherited an h4 that should have been h3. Fixing it in my own files worked as a stop-gap and I flagged it to the team for a proper shared fix.

The spelling mistake in my CSS ("balck" instead of "black") was actually the funniest one to find - the validator caught it as an unknown value which is exactly what a strict CSS check does. Made me realise CSS validation matters as much as HTML validation. From now on I run both after every big edit.
