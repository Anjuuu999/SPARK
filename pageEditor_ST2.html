Splash Screen

Technical Description

For the Splash Screen I built a standalone landing page that acts as the first impression of the SPARK site. The page does not use the shared template because the coursework brief says the splash should be its own thing. I used semantic HTML5 to structure it - the main content is inside a <div class="splash-container"> that holds the logo image, a page title, the mission text, our team members, a CSS loader, a countdown paragraph and a Skip Intro link. The redirection to home page happens through a <meta http-equiv="refresh" content="4; url=home.html"> tag in the head, which is triggered automatically after 4 seconds.

On the styling side, I applied a dark linear gradient overlay on top of a background image to keep the white text readable no matter what the photo behind is. I centered everything using flexbox (display: flex, align-items: center, justify-content: center) which was cleaner then using absolute positioning like I tried first. The loader animation is entirely CSS - I used a <div class="loader"> element with three transparent borders and one green top border, spun using a @keyframes spin animation with animation: spin 1s linear infinite. The brief specifically banned GIFs so making it in pure CSS was a requirement.

The countdown is powered by JavaScript. I used setInterval that fires every 1000ms and decrements a count variable from 4. The updated number is written back into <span id="count"> using textContent. When count hits 0 the interval is cleared using clearInterval() so it doesn't keep running in the background. The Skip Intro button is a normal anchor tag pointing to home.html which lets people bypass the wait if they want.

Accessibility

All text sits above the dark overlay layer so contrast stays good against any image behind it. I gave the logo image proper alt text describing what it is. The Skip Intro link is a real anchor element not a fake button styled as one, which means keyboard users can Tab straight to it and screen readers pick it up as a link. The h1 gives the page a clear document title for assistive tech. I also kept the redirect delay at 4 seconds which is enough to read the tagline without feeling rushed but not so long that keyboard users get stuck.

Action Impact Simulator

Technical Description

The AIS is my main interactive page and probably where most of my JavaScript marks come from. The page uses the shared template with the standard header, main and footer. Inside main, I used a <section> for the intro hero and then a grid layout for the 6 climate action cards. Each card is a <div class="action-card"> with a data-points attribute holding the score value, plus an image, a title, a description and a small points display. Below the cards there's a feedback panel that shows count, total, level and a message.

For the CSS I used CSS Grid to split the cards into two rows - the top row has 2 large cards using grid-template-columns: 1fr 1fr and the bottom row has 4 smaller cards using repeat(4, 1fr). The selected state adds a bright green border, lifts the card up with transform: translateY(-4px) and shows a small tick mark using the ::after pseudo element. I also added media queries so on smaller screens the grid collapses to 2 columns and then 1 column so cards don't squash together. The background image on the body swaps between three photos depending on the impact level - this is done through JavaScript setting document.body.style.backgroundImage.

The JavaScript is split into three short functions. toggleCard(this) is called on click and it flips the selected class on the card that was clicked, then calls updateFeedback(). updateFeedback() uses querySelectorAll('.action-card.selected') to find all the currently selected cards, sums up their data-points values (wrapped in Number() because I hit a bug where the values were being concatenated as strings instead of added), and writes the total to the score element. Then an if/else if chain decides the impact level - Low (0-6), Medium (7-13) or High (14+) - and updates the label, message and body background. The reset button uses resetSimulator() which iterates every card with forEach and removes the selected class.

Accessibility

All six card images have proper alt text describing the action, not just the object. Cards have a clear visible selection state with the bright green border and tick mark. The hover lift also gives feedback that the card is interactive. Heading hierarchy runs h1 to h2 to h3 in order. The feedback panel updates right after each click so users always see the current impact level - useful for screen reader users too. One thing I know could be improved is the cards use div elements with onclick handlers - if I had more time I would swap them for <button> elements or add role="button" with tabindex="0" so keyboard-only users can activate them by pressing Enter or Space.

Content Page - Transport & Travel

Technical Description

The Content Page covers Transport and Travel which is my angle within SDG 13 Climate Action. I structured it using semantic HTML5 - the main content is wrapped in a <main> with a hero section at the top, then a <nav class="jump-nav"> for internal navigation, then 5 content sections. Each content section is a proper <section> element with a unique id (like #scale, #commute, #flights) so the jump-nav pill links can jump directly to it. Heading structure is strict - h1 in the hero, h2 for each section, h3 in the footer.

For CSS I used flexbox on the hero to center everything and I put a dark linear gradient overlay on top of the background image so the white h1 reads properly. Each content section has scroll-margin-top of 120px so when someone clicks a jump-nav pill the target heading doesn't end up hidden behind the fixed header - this took me a while to figure out because at first the jumps were landing but I couldn't see the heading. Every content section header has a small green left border which gives visual rhythm without needing dividers. Images are styled with border-radius, box-shadow, responsive max-width and a bit of margin so they look like proper article images.

The Go to Top button in the bottom right corner is done with pure CSS - position: fixed; bottom: 30px; right: 30px and an anchor pointing to the body id="top". No JavaScript needed for it. The whole content page works even without JavaScript because everything runs on HTML anchors and CSS - which was a nice thing to realise because it means the page loads instantly and stays fully useable even if scripts fail.

Accessibility

Every content image has meaningful alt text describing whats in it. Heading hierarchy is h1 to h2 to h3 in order with no skips. The jump-nav has an aria-label="Page sections" so screen readers announce it as a landmark. The Go to Top button has aria-label="Back to top". Contrast between the dark green text (#074534) on light backgrounds is comfortably within WCAG AA. External reference links use target="_blank" so users don't lose where they were reading. I grouped all citations into the References section on the Page Editor rather than dumping them at the bottom of the content page which keeps the reading flow clean.

Challenges and Lessons Learned

The biggest challenge with the Splash Screen was actually a really small thing - my countdown was working in the JavaScript variable but the number on the screen was frozen. I spent maybe half an hour debugging before I realized I was decrementing the count variable but forgetting to write it back to the DOM with textContent. Once I added that line the countdown worked perfectly. Small mistake but it taught me to always check the render path when JS state changes.

For the AIS, my first version had six separate onclick handlers - one per card. It worked but was ugly and adding a seventh action would have been copy-paste hell. I refactored it into a single toggleCard(this) function that reads each card's data-points attribute. Cut the code roughly in half. Later I hit another bug where selecting cards was updating the total wrong - it was appending strings instead of summing. Wrapping the read in Number() fixed it. Two small bugs but they both taught me valueable lessons about JavaScript type coercion and DRY principles.

For the Content Page, the jump-nav pills were jumping to the right anchor but the target heading was always hidden behind the fixed header. Turns out this is what scroll-margin-top is designed to solve - I added 120px on every content section and the offset worked cleanly. Same issue on the hero at first (the h1 was covered by the header on page load) which I fixed by increasing the hero's padding-top to 130px. Now I use these tricks by default whenever there's a fixed header sitting above anchor-based navigation.

Overall lesson - small typos or missing lines can silently break stuff that looks like it should just work. I learnt to check three things when a page misbehaves: the browser console for errors, the elements panel for wrong or missing classes and the network tab for failed asset loads. Most of my bugs were one of those three.

Accessibility Considerations

Across all my three pages I made sure accessibility was baked in from the start rather then bolted on at the end. Here's what I did:

  • Every image has descriptive alt text - not just "image" but a proper description of whats being shown.
  • Semantic HTML5 elements are used everywhere: <header>, <nav>, <main>, <section>, <footer>.
  • Heading hierarchy runs h1 to h2 to h3 in order on every page with no skips.
  • The Content Page jump navigation uses aria-label="Page sections" and the Go to Top button uses aria-label="Back to top".
  • Colour contrast between white text (#ffffff) on the dark forest backgrounds (#074534, #051b11) meets WCAG AA guidelines.
  • Interactive elements like the Skip button and nav links are proper anchor or button elements so they're keyboard accessible by default with visible focus states.

Compliance with JANET Regulations

All pages I created for SPARK follow JANET regulations for responsible educational web publication. The following principles were applied:

  • Original content: Everything written on my three pages is my own words. No copy-paste from the UN SDG site or any other source.
  • Image sourcing: All images come from royalty-free platforms (Vecteezy, Unsplash, Piqsels, Wallpaperflare) that permit non-commercial educational use. Full attribution is in the References section below.
  • No personal data collection: The site doesn't collect, store or transmit any personal user data. No cookies, no local storage, no trackers.
  • Appropriate content: All content is educational and aligned with SDG 13. Nothing offensive, discriminatory or harmful.
  • Academic integrity: All HTML, CSS and JavaScript on my pages was written by me from scratch. No pre-built templates or copied code.
  • Accessibility compliance: Pages follow WCAG accessibility principles including semantic HTML, alt text and keyboard navigation.
