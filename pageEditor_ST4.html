User Profile Page

Technical Description

The User Profile page (profile.html) was built using the shared template structure with the global header, navigation bar, and footer. The page uses the <main> semantic tag to wrap the core interactive area, which contains a progress bar and a profile output section.

The HTML structure consists of a <div class="profile-container"> that holds three hidden <div class="profile-section"> elements (one for each step: Personal Details, Interests, and Commitments). Each section has a unique id attribute (e.g., id="section-personal") so JavaScript can target and reveal them individually using document.getElementById().

The progress bar is implemented with two nested <div> tags: an outer container with a grey background and an inner bar whose width is dynamically set by JavaScript. A CSS transition: width 0.5s ease-in-out property creates a smooth animated fill effect whenever the progress updates.

Interactivity is entirely driven by JavaScript using prompt() dialogs. The script defines an array of step objects, each containing an array of prompt questions. When the user clicks "Start / Continue Profile," the startProfileBuilder() function iterates through the current step's prompts. User responses are stored in a userData object. After each step completes, the updateProfileOutput() function re-renders the visible sections by setting innerHTML with template literals and toggling style.display between 'block' and 'none'.

Users can cancel a prompt (returning null) to pause the step, leave a prompt blank to skip it, or use the "Revisit Step" buttons to return to any previously completed step. A resetProfile() function clears the userData object and resets the step index to zero.

CSS styling uses embedded <style> in the <head> section. The navigation bar's active state is handled by overriding the global CSS nav span position to move the sliding indicator to the fourth link, and using attribute selectors such as nav ul li a[href="profile.html"] to set the "Profile" link colour to greenyellow, without modifying the shared style.css file.

Accessibility

The page uses semantic HTML tags (<header>, <main>, <footer>) ensuring screen readers can parse the document hierarchy. All buttons have descriptive text labels ("Start / Continue Profile," "Reset Profile," "Revisit Step 1/2/3") so users relying on assistive technology understand the purpose of each control. The progress bar includes a text-based percentage readout (id="progress-text") so the completion status is conveyed to non-visual users, not just through the visual bar width.

Sitemap Page

Technical Description

The Sitemap page (sitemap.html) presents a visual, interactive map of the entire website hierarchy using Scalable Vector Graphics (SVG). The core of the page is a single <svg> element with a viewBox="0 0 1000 600" and preserveAspectRatio="xMidYMid meet", making the diagram fully responsive across different screen sizes without distortion.

The site structure is represented using SVG shape primitives: a <circle> for the Home page (the central hub) and <rect> elements with rounded corners (rx attribute) for all other pages. These shapes are grouped inside <g class="node"> elements, each containing an <a> tag wrapping the shape and a <text> label. The <text> elements use text-anchor="middle" for centred alignment within their respective shapes.

Connecting lines between nodes are drawn using SVG <path> elements with the d attribute defining line coordinates (e.g., d="M 500 60 L 500 120"). These paths use the class "link" and are styled with a neutral grey stroke.

CSS hover and focus effects are applied via embedded styles targeting .node:hover rect and .node a:focus rect, changing the fill colour and stroke width to provide clear visual feedback. The pointer-events: none property on <text> elements ensures mouse clicks pass through the label to the underlying clickable shape.

Accessibility

Each clickable SVG node contains a <title> element (e.g., <title>User Profile Page</title>) providing descriptive tooltips for screen readers. All interactive <a> elements within the SVG have tabindex="0" to ensure full keyboard navigation. The :focus pseudo-class provides visible focus indicators (colour and stroke changes) matching the hover state, so keyboard-only users receive equivalent visual feedback. The overall <svg> tag includes an aria-label attribute describing the diagram's purpose.

Content Page

Technical Description

The Content page (content_ST4.html) is an informational article titled "Household Energy Consumption and Its Direct Climate Impact." The page layout uses CSS Flexbox (display: flex; flex-direction: row; gap: 30px) to position a sticky sidebar navigation alongside the main article body.

The sidebar is implemented as a <div class="side-nav" role="navigation"> rather than a <nav> element. This deliberate choice was made because the style.css file applies position: absolute; left: 50%; transform: translateX(-50%) to all <nav> elements (intended for the header navbar). Using a <div> with role="navigation" preserves accessibility semantics while avoiding the inherited absolute positioning that would break the sidebar layout.

The sidebar uses position: sticky; top: 110px so it remains visible as the user scrolls through the article. It contains anchor links (<a href="#fossil-fuels">, etc.) that scroll smoothly to corresponding <section> elements via the CSS property html { scroll-behavior: smooth }.

The article body is divided into four <section> elements, each with a unique id, containing structured headings (<h1>, <h2>, <h3>) and four locally stored images within the ./images/ directory. Each <img> tag has a descriptive alt attribute and uses the .content-image class for consistent styling (border-radius, box-shadow, and responsive width).

The global main styles are overridden using a specific class selector main.content-page to set display: block and adjust padding, ensuring the Flexbox layout of the .content-wrapper div works correctly without conflicting with the global main { flex: 1; display: flex } rule.

A fixed "Go to Top" button is implemented as a pure HTML/CSS anchor link (<a href="#top" class="back-to-top">) positioned with position: fixed; bottom: 30px; right: 30px. No JavaScript is required for this functionality.

Accessibility

All four images include descriptive alt text that conveys the content of each image (e.g., "Photovoltaic solar panels mounted securely across the shingled roof of a suburban family house"). The sidebar uses role="navigation" and aria-label="Page Sections" to identify its purpose for screen readers. The heading hierarchy (h1 > h2 > h3) is maintained correctly throughout the page. The "Go to Top" button includes an aria-label="Scroll back to top of the page" attribute.

Challenges and Lessons Learned

Challenge 1 - Global CSS conflicts with the sidebar navigation: The most significant challenge was on the Content page. The style.css applies position: absolute and transform: translateX(-50%) to all <nav> elements. When I initially used a <nav> tag for my sidebar, it flew off the page and centred itself over the header. I resolved this by changing the sidebar to a <div> with role="navigation" and explicitly overriding the global nav ul styles (like display: flex) within the embedded CSS. This taught me the importance of understanding CSS specificity and how shared stylesheets can have unintended side effects on page-specific elements.

Challenge 2 - Progressive DOM updates in the User Profile: Managing the step-by-step data collection via prompt() while keeping the UI in sync was tricky. Initially, the DOM would duplicate data if a user revisited a step. I solved this by using a central userData object as the single source of truth and having the updateProfileOutput() function completely re-render all visible sections based on the current state of that object, rather than appending new elements.

Challenge 3 - SVG coordinate alignment: Manually positioning SVG <rect>, <circle>, <text>, and <path> elements required careful calculation of x/y coordinates. I learned how the SVG viewBox coordinate system works independently of the page's CSS dimensions, and how preserveAspectRatio handles scaling.

Challenge 4 - Active navigation state without editing global CSS: The shared style.css hardcodes the first nav link as the active/highlighted item. To show "Profile" as active on my page, I used embedded CSS with higher specificity to override the default nav span position and link colours. This approach respects the group workflow by not modifying the shared file.

Compliance

All content on my three pages is original, written by me from my own research. Where I referenced external facts and statistics, I cited the original source in-text and included a full entry in the References section. No copyrighted articles, images, or code have been reproduced verbatim.

The images used on my pages come from either the SPARK team's own image folder or from stock sources that permit non-commercial educational use. Any external image sources are attributed in the References section. No branded logos, trademarks, or celebrity likenesses appear anywhere on my pages, which keeps the project on the safe side of Janet acceptable use guidelines.

The site collects no personal data, uses no third-party trackers, and stores nothing in cookies or local storage. All interactivity happens client-side within the browser session and disappears when the user closes the tab. This means the project sits comfortably within data protection expectations for a student educational project.

Accessibility has been treated as a compliance requirement rather than a nice-to-have. Semantic HTML tags, descriptive alt text on every image, a strict heading hierarchy, ARIA labels on non-obvious landmarks, and adequate contrast ratios all sit within the Janet expectations for public-facing educational content and are consistent with the WCAG guidance covered in the module lectures.

Nothing on the pages is offensive, defamatory, or promotes illegal activity. The tone is educational and constructive, aligned with the SDG 13 climate action theme.
