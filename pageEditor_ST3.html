Feedback page

Technical Description

I did use semantic HTML5 tags to create a clean and accessible structure for the Feedback page. The content of the page is grouped into the main content of the page using a <main> tag, and into logical sections using <section> elements for the program directory, the review summary, and for the input form itself. On the feedback form, I used the <fieldset> tag to create groups of related fields and <legend> tag to include descriptions of the form fields for assistive technologies. I selected a type of "text", "email", "tel", "date" and "datetime-local" for the input types and included a "textarea" for the custom message, so that the user is asked only for the correct data types from the beginning.

On the presentation layer, I utilized cutting-edge CSS skills to develop a layout that is attractive and orderly. To keep it simple and usable while still allowing the program cards to roll horizontally, I used Flexbox and the overflow-x: auto property. I used smooth transitions and hover behaviour for the cards and submission buttons so that they would 'jump' or change colour slightly when a user interacted with them to emphasise interactivity. Colour and spacing were organised and the layout is clean and easy-to-read for all screen sizes.

The last layer of the page is the interactive JavaScript logic that is used to enhance the user experience and perform client-side validation on the form before it is sent. I added a real-time character counter activated by the text area's input event listener that dynamically updates the number of characters that have been entered into the text area and shows the number of characters remaining. The script catches the submit event, prevents the page from refreshing, and iterates through all the fields checking if they are blank, or if the email and the telephone field do not satisfy certain pattern. If an error is found, then the script draws a red outline around the particular field of the form, and an error message appears immediately below the field; if no errors are found, then a green success message banner is drawn, and the form is reset completely.

Accessibility

I have made a self-guided effort to make the Feedback page accessible and inclusive for all, from the beginning. I applied semantics with HTML5 elements such as <main>, <section> and <header> to provide an assistive technology with a clear path through the document. The hierarchy of the headings is logical; that is, there are no missing headings between <h1> and <h4>; screen readers can give an outline of the headings. The interactive features were done with a <fieldset> and <legend> to organize the form controls properly, and by adding a for attribute on the <label> element to match the input element it presents, to design the interactive elements like labels to connect with the input elements. And by applying for attribute on the <label> element to match the <input> element it presents, to design the interactive elements like <label> elements to connect with its input element. Also added role="status" attribute on the confirmation message to announce the changes immediately. The images found in program thumbnails, for example, have descriptive alt text to communicate the meaning of the image to a visually impaired user. A current restriction, however, is that the program directory scrolls sideways and is too mouse-heavy - the next step will be making a program even more keyboard friendly by adding arrow-key support to fully keyboard enable scrolling.

Team page

Technical Description

For formatting the Team page I used the appropriate HTML5 elements in semantic HTML5, so that it is structured and laid out properly to create a clear and accessible hierarchy. The structure of the layout is based on <main> element that contains the main content of a web page and introduces a <section> element that will contain the introductory content. The group members are enumerated with individual <article> tags and the special tabindex="0" attribute. This design choice helps to make every team profile an interactive card widget that is easily recognized by screen readers and keyboard users. The use of text sizes is logical, with <h1> for the title and then <h3> for the member names for each section, creating a well-structured and easy-to-navigate outline. Within each profile, further-nested lists are implemented with <ul> and <li> tags, to ensure that particular contributions to coursework are clearly distinguished.

The styling and presentation required a strong CSS Grid layout with grid-template-columns: repeat(2, 1fr) to partition the screen into a balanced, two column layout. This page relies on advanced CSS selector combinations such as :hover, :focus and :focus-within to provide all of the interactive transitions without JavaScript. A user clicking on one of the cards causes the profile picture to grow larger in size and the <div> with additional details to expand smoothly with max-height. I also added a unique :focus-visible state to protect keyboard accessibility by adding a bold accent outline so that everyone will have the same experience with focus as they will have using their eyes.

Accessibility

Throughout the development of this page, I worked to make it accessible and inclusive for all. Semantic HTML5 elements such as <main>, <section> and <article> were used to provide a clear structure for assistive technologies. The heading hierarchy is logical from <h1> to <h3> with no missing levels and provides a good outline for screen readers. The tabindex is practically explicitly assigned to each team card, thus creating team cards that can be focused and navigated by keyboard users. Images, such as profile pictures, have descriptive alt text to provide context for visually impaired users and a unique :focus-visible outlined appearance when using the keyboard.

Content page

Technical Description

I used semantic HTML5 tags to organize information efficiently and properly for standard visitors and assistive technologies to make the content page of Nature & Green Spaces easy to navigate. The overall scope, or structural scope, for the whole core topic is defined by a <main> wrapper. An introductory paragraph sets the context before a secondary <nav> block offers jump links for navigating within the page. The main informational concept is broken down into individual blocks of <section> elements that are each semantically identified (e.g., forests or urban-green). There will be a consistent hierarchy of headings, with the main topic represented by an <h1> tag, and sub-topics within each section of the layout will be represented by an <h2> tag, providing an easy-to-navigate roadmap for the entire document.

I made a clean information flow with the core grid, using CSS Flexbox rules to layout and present it in a column layout (display: flex; flex-direction: column). By default, each individual content card is presented horizontally, but I added flex-direction: row-reverse to every content card with an even index (i.e. <section index="2">, <section index="4">, etc.) to create an interesting alternating layout. Responsiveness is achieved using a CSS media query to change the flex-direction to column when the width/size of the screen is below 700px through the interface. Smooth transitions linked to the :hover and :focus-within values bring about cards to life with the help of translateY(-6px) and by introducing a visible highlight border, the presentation changes in real time, without forcing client side scripts to execute.

Accessibility

Throughout the development of me, I have made sure the Nature & Green Spaces content page is inclusive and accessible to all digitally. Incorporated semantic HTML5 elements such as <main>, <section>, and <nav> to provide an assistive technology a document outline. The hierarchy of headings is logical, starting from an <h1> and continuing to an <h2> subheading and not missing a level, which is helpful for the screen reader to properly outline the content. To create the interactive layout, I used explicit id attributes on sections and an in-page navigation block with an aria-label="Page sections" which allows a keyboard user to quickly skip to a topic. Passive layouts are made focusable with the addition of a tabindex="0" and decorative and context-rich images are accompanied by descriptive alt text so that visually impaired users can follow along easily.

Challenges and Lessons Learned

The main issues I had to overcome during development was the global stylesheet file 'style.css' and the local style sheet, which I would have to deal with unexpected style inheritance and specious selector conflicts. If the global file targeted the raw <nav> tag with the high-priority placement rules that only apply to the main header, for example when designing the page-specific jump navigation menus, then it should not be used. This made my sub navigation bars to center absolutely, move out of place, and disappear right behind the main navigation menu. I started to try to work around these restrictions with ill-formed cascading sets of !important styles, but that soon made my stylesheet messy. The most efficient way to do this was to refactor the html, using a generic <div> with an own class name and completely removing it from the global header "scope.

A related concrete layout problem arose in implementing the interactive card systems on both the Team and the Content pages. Use smooth transform: translateY(-6px) with a box-shadow above the box, to raise the cards dynamically. This was fine for some of the blocks, but the flexbox block-configuration cards didn't budge at all, even when they were supposed to hover based on the state of the link. While researching the inner workings of the layout engines of browsers, I discovered that sometimes, elements with dynamic flex behavior do not have an explicit positioning baseline to other structural siblings. This was fixed by explicitly setting position: relative; on the base class of the card, which instantly corrected the layout context and made all the grid items position: relative; and move at the same rate.

Also, some image links were broken in the initial deployment, due to managing asset paths with a multi-page directory structure. Relative paths such as images/team-krishen.jpg were causing some failures while moving between roots, depending on the nesting of the pages in the local development environments. I had to do absolute file consistency within each HTML document and correct all the folder references, which meant that I had to go through the whole folder structure and audit it. Meanwhile, I saw that large image asset sizes were causing text blocks within card flex boxes to get skewed. I fixed this presentation issue by using object-fit: cover and hard coded the width constraints so that all photos were cropped so that they fit perfectly without changing their aspect ratios.

In the end this development process taught me the crucial need to take the time to write code that is modular, defensive and plan out layouts before styling them. I learned that making use of common element selectors like <nav> and <p> in global stylesheets is a fragile codebase because, down the road, it can cause unexpected style leaks to break the independent parts of the application. But I needed to change to explicit class names to resolve my navigation problems, and to make my files much easier to read. Also, by using CSS alone instead of JavaScript, (with some advanced pseudo-classes) to create a dynamic, interactive system, it demonstrated the power of semantic HTML and well-used pseudo-classes. In the future, I will focus on ensuring that my websites are scalable, clean and fully functional - from the outset I will think about the structure and layout baseline, testing it across browsers and planning for accessibility.

Compliance

All content on my three pages is original, written by me from my own research. Where I referenced external facts and statistics, I cited the original source in-text and included a full entry in the References section. No copyrighted articles, images, or code have been reproduced verbatim.

The images used on my pages come from either the SPARK team's own image folder or from stock sources that permit non-commercial educational use. Any external image sources are attributed in the References section. No branded logos, trademarks, or celebrity likenesses appear anywhere on my pages, which keeps the project on the safe side of Janet acceptable use guidelines.

The site collects no personal data, uses no third-party trackers, and stores nothing in cookies or local storage. All interactivity happens client-side within the browser session and disappears when the user closes the tab. This means the project sits comfortably within data protection expectations for a student educational project.

Accessibility has been treated as a compliance requirement rather than a nice-to-have. Semantic HTML tags, descriptive alt text on every image, a strict heading hierarchy, ARIA labels on non-obvious landmarks, and adequate contrast ratios all sit within the Janet expectations for public-facing educational content and are consistent with the WCAG guidance covered in the module lectures.

Nothing on the pages is offensive, defamatory, or promotes illegal activity. The tone is educational and constructive, aligned with the SDG 13 climate action theme.
